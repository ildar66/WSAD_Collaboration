Impostazioni JVM (Java Virtual Machine)

Utilizzare questa pagina per visualizzare e modificare la configurazione JVM (Java Virtual Machine) per il processo del server delle applicazioni.

Per visualizzare questa pagina della console di gestione, fare clic su Server > Server delle applicazioni > nome_server > Definizione processo > JVM (Java Virtual Machine).

Scheda Configurazione

Classpath
Specifica il percorso classe standard in cui il codice Java virtual machine ricerca le classi.

Immettere ciascuna voce del percorso classe in una riga della tabella. Non è necessario aggiungere i due punti o il punto e virgola alla fine di ogni voce.

Dimensione heap iniziale
Specifica la dimensione heap iniziale disponibile per il codice JVM, in megabyte.

L'aumento delle dimensioni minime dell'heap può migliorare l'avvio. Il numero di occorrenze di raccolta dati inutili viene ridotto e le prestazioni vengono ottimizzate del 10%.

In generale, l'aumento delle dimensioni heap Java migliora la velocità di trasmissione fino a quando l'heap non si trova più nella memoria fisica. Dopo che l'heap inizia ad essere trasferito sul disco, le prestazioni Java ne risentono drasticamente.

Tipo dati Numero intero
Predefinito 64 per OS/400, 50 per tutte le altre piattaforme
Dimensione massima heap
Specifica la dimensione heap massima disponibile per il codice JVM, in megabyte.

L'aumento delle dimensioni dell'heap può migliorare l'avvio. Il numero di occorrenze di raccolta dati inutili viene ridotto e le prestazioni vengono ottimizzate del 10%.

In generale, l'aumento delle dimensioni heap Java migliora la velocità di trasmissione fino a quando l'heap non si trova più nella memoria fisica. Dopo che l'heap inizia ad essere trasferito sul disco, le prestazioni Java ne risentono drasticamente. Quindi, impostare la dimensione massima di heap a un valore sufficientemente basso per contenere l'heap nella memoria fisica.

Tipo dati Numero intero
Predefinito 0 per OS/400, 256 per tutte le altre piattaforme. Mantenere il valore sufficientemente basso per evitare lo spostamento o swapping dei dati dalla memoria su disco.

Argomenti JVM generici
Specifica gli argomenti della riga comandi da trasferire al codice Java Virtual Machine che avvia il processo del server delle applicazioni.

Di seguito vengono riportati gli argomenti facoltativi della riga comandi che è possibile utilizzare immettendoli nel campo Argomenti JVM generici:

  • -Xquickstart: è possibile utilizzare questo valore per la compilazione iniziale ad un livello di ottimizzazione inferiore rispetto alla modalità predefinita e, successivamente, a seconda dei risultati di esempio, è possibile compilare di nuovo al livello della compilazione iniziale in modalità predefinita. Utilizzare l'avvio rapido per le applicazioni in cui la velocità moderata iniziale è più importante della velocità di trasmissione costante durante l'esecuzione. In alcuni scenari di debug, strumenti di test e strumenti che richiedono tempi di esecuzione brevi, è possibile migliorare il tempo di avvio del 15-20%. -DCOPT_NQREACHDEF consente di migliorare l'avvio di un ulteriore 15%.
  • -Xverify:none: quando si utilizza questo valore, la fase di verifica di classe viene ignorata durante il caricamento della classe. Utilizzando -Xverify:none con il compilatore JIT (just in time) abilitato, il tempo di avvio viene migliorato del 10-15%.
  • -Xnoclassgc: è possibile utilizzare questo valore per disabilitare la raccolta dati inutili, rendendo più disponibile il riutilizzo della classe e migliorando leggermente le prestazioni. La raccolta dati inutili viene abilitata per impostazione predefinita, ma si consiglia di abilitare questa funzione. È possibile monitorare la raccolta dati inutili mediante l'impostazione di configurazione verbose:gc, poiché l'output include le statistiche raccolta dati inutili della classe.
  • -Xgcthreads: è possibile utilizzare contemporaneamente diversi thread raccolta dati inutili, noti anche come raccolta dati inutili parallela. Quando si immette questo valore nel campo Argomenti JVM generici, immettere anche il numero dei processori della macchina, ad esempio -Xgcthreads=numero_di_processori. Si consiglia di utilizzare la raccolta dati inutili parallela se la macchina dispone di più di un processore. Questo argomento si applica solo a IBM Developer Kit.
  • -Xnocompactgc: questo valore disabilita la compressione heap che è l'operazione raccolta dati inutili più dispendiosa. Evitare la compressione in IBM Developer Kit. Se si disabilita la compressione heap, eliminare tutti i sovraccarichi associati. Quando si immette questo valore nel campo Argomenti JVM generici, immettere anche il numero dei processori della macchina, ad esempio -Xnocompactgc=numero_di_processori.
  • -Xinitsh: è possibile utilizzare questo valore per impostare la dimensione iniziale dell'heap in cui vengono memorizzati gli oggetti di classe. Le definizioni del metodo e i campi statici vengono inoltre memorizzati con gli oggetti di classe. Sebbene le dimensioni dell'heap di sistema non presentino un limite superiore, impostare la dimensione iniziale in modo da non comportare dei costi di espansione delle dimensioni dell'heap di sistema, che richiedono chiamate al gestore della memoria del sistema operativo. È possibile calcolare una buona dimensione iniziale dell'heap di sistema se si conosce il numero delle classi, circa 8000, caricate nel prodotto WebSphere e le relative dimensioni medie. Conoscendo le guide dell'applicazione, includerle nel calcolo.
  • -Xmc: la dimensione heap locale del thread è una parte dell'heap assegnata esclusivamente a un thread. A causa della dimensione heap locale del thread, non è necessario che il thread blocchi l'intero heap durante l'allocazione degli oggetti. Tuttavia, quando l'heap locale del thread è pieno, l'allocazione degli oggetti viene eseguita dall'heap che deve essere sincronizzato. Una buona dimensione della cache locale è cruciale per le prestazioni e richiede di conoscere l'applicazione e i relativi oggetti.
  • -Xml: è possibile utilizzare questo valore per impostare il limite della dimensione di un oggetto da allocare dalla cache locale. Gli oggetti la cui dimensione supera il limite devono essere allocati nell'heap regolare. Allocare gli oggetti dalla cache locale per quanto possibile oppure la cache locale viene scaricata poiché non aumenta in modo dinamico. Se alcuni oggetti diventano molto grandi, allocarli dall'heap regolare.
