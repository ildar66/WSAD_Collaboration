Einstellungen für die Anwendungsinstallation vorbereiten

Auf dieser Seite können Sie eine Anwendung (EAR-Datei) oder ein Modul (JAR- oder WAR-Datei) installieren.

Klicken Sie, um diese Seite der Administrationskonsole anzuzeigen, auf Anwendungen > Neue Anwendung installieren.

Führen Sie die Schritte auf dieser Seite durch, um eine Anwendung oder ein Modul zu installieren. Sie müssen zumindest den ersten Schritt durchführen. Ob Sie einige oder alle nachfolgenden Schritte durchführen, ist abhängig davon, ob Sie eine Anwendung, ein EJB-Modul oder ein Webmodul installieren:

Pfad
Gibt den vollständig qualifizierten Pfad zur .ear-, .jar- oder .war-Datei für die Enterprise-Anwendung an.

Verwenden Sie Lokaler Pfad, wenn der Browser und die Anwendungsdateien auf derselben Maschine installiert sind (unabhängig davon, ob der Server auf der Maschine installiert ist).

Verwenden Sie Serverpfad, wenn die Anwendungsdatei sich auf einem Knoten im aktuellen Zellenkontext befindet. Sie können das gesamte Dateisystem eines Knotens durchsuchen, wenn ein Knotenagent oder Deployment Manager auf diesem ausgewählten Knoten ausgeführt wird. Beim Durchsuchen werden nur .ear-, .jar- oder .war-Dateien angezeigt.

Während der Anwendungsinstallation werden die Anwendungsdateien in der Regel von einer Clientmaschine, auf der der Browser ausgeführt wird, auf die Servermaschine, auf der die Administrationskonsole ausgeführt wird, hochgeladen. Auf dieser Servermaschine wird das Deployment der Anwendungsdateien durchgeführt. In diesen Fällen werden die auf die Servermaschine hochzuladenden EAR-, WAR- und JAR-Module mit dem Webbrowser ausgewählt, in dem die Administrationskonsole ausgeführt wird.

In manchen Fällen jedoch befinden sich die Anwendungsdateien bereits im Dateisystem eines der Knoten in einer Zelle. Soll der Anwendungsserver diese Dateien installieren, müssen Sie die Option Serverpfad verwenden.

Sie können diese Option auch verwenden, um eine Anwendungsdatei anzugeben, die sich bereits auf der Maschine, auf der der Anwendungsserver ausgeführt wird, befindet. Ein Beispiel für den in das Feld einzutragenden Wert unter Windows NT ist C:\WebSphere\AppServer\installableApps\test.ear. Wenn Sie ein eigenständiges WAR-Modul installieren, müssen Sie auch den Stammkontext angeben.

Stammkontext
Gibt den Stammkontext der Webanwendung (WAR) an.

Dieses Feld wird nur zur Installation einer eigenständigen WAR-Datei verwendet. Der Stammkontext wird mit der definierten Servlet-Zuordnung (aus der WAR-Datei) zum vollständigen URL kombiniert, den die Benutzer eingeben müssen, um auf das Servlet zugreifen zu können. Beispiel: Wenn der Stammkontext /gettingstarted ist und die Servlet-Zuordnung MySession lautet, ergibt sich daraus der URL http://host:port/gettingstarted/MySession.

Standardbindungen generieren
Gibt an, ob Standardbindungen generiert werden sollen. Wenn Sie das Markierungsfeld auswählen, werden für alle unvollständigen Bindungen in der Anwendung Standardwerte angegeben. Vorhandene Bindungen werden nicht geändert.

Durch Auswahl dieser Option können Sie direkt zum Schritt "Zusammenfassung" gehen und die Anwendung installieren, vorausgesetzt, keiner der Schritte ist mit einem roten Stern (*) gekennzeichnet. Ein roter Stern gibt an, dass der entsprechende Schritt unvollständige Daten hat und einen gültigen Wert benötigt. Prüfen Sie in der Anzeige "Zusammenfassung" die Zelle, den Knoten und den Server, auf dem die Anwendung installiert ist.

Bindungen werden wie folgt generiert:

  • EJB-JNDI-Namen werden im Format Präfix/ejb-Name generiert. Das Standardpräfix ist ejb, es kann jedoch überschrieben werden. Der ejb-Name lautet wie im <ejb-name>-Tag des Deployment-Deskriptors angegeben.
  • EJB-Referenzen werden wie folgt gebunden: Wenn ein <ejb-link> ermittelt wird, wird das berücksichtigt. Wird eine eindeutige Bean mit einem übereinstimmenden Home- oder Local-Home-Interface als referenzierte Bean ermittelt, wird die Referenz automatisch aufgelöst.
  • Bindungen der Ressourcenreferenzen werden vom <res-ref-name>-Tag abgeleitet. Beachten Sie Folgendes: Es wird vorausgesetzt, dass der java:comp/env-Name mit dem globalen JNDI-Namen der Ressource identisch ist.
  • Bindungen von Verbindungs-Factorys (für EJB-2.0-JAR-Dateien) werden auf der Grundlage des JNDI-Namens und der Berechtigungsangaben generiert. Dies führt dazu, dass die Einstellungen der Standard-Verbindungs-Factory für jede EJB-2.0-jar-Datei in der zu installierenden Anwendung definiert werden. Es werden keine Bindungen von Verbindungs-Factorys auf Bean-Ebene generiert.
  • Bindungen von Datenquellen (für EJB-1.1-JAR-Dateien) werden auf der Grundlage des JNDI-Namens und der Kombination aus Benutzername und Kennwort der Datenquelle generiert. Dies führt dazu, dass für jede EJB-JAR-Datei Einstellungen der Standarddatenquelle definiert werden. Es werden keine Bindungen von Datenquellen auf Bean-Ebene generiert.
  • MDB-Listener-Ports (Message Driven Bean, nachrichtengesteuerte Bean) werden vom <ejb-name>-Tag des MDB, an den die Zeichenfolge Port angehängt ist, abgeleitet.
  • Bei .war-Dateien wird der virtuelle Host als Standard-Host definiert, wenn nicht anders angegeben.

Das Standardverfahren ist für die meisten Anwendungen oder zumindest für die meisten Bindungen in den meisten Anwendungen ausreichend. In folgenden Fällen ist das Verfahren jedoch nicht anwendbar:

  • Sie möchten die globalen JNDI-Namen einer oder mehrerer EJB-Dateien explizit steuern.
  • Sie benötigen eine stärker unterteilte Steuerung der Bindungen von Datenquellen für CMPs. Das bedeutet, Sie haben mehrere Datenquellen und benötigen mehrere globale Datenquellen.
  • Sie müssen Ressourcenreferenzen globalen JNDI-Namen von Ressourcen, die nicht mit dem java:comp/env-Namen identisch sind, zuordnen.

In diesen Fällen können Sie das Verhalten mit einem XML-Dokument (benutzerdefinierte Strategie) ändern. Sie können im Feld Spezifische Bindungsdatei eine benutzerdefinierte Strategie angeben und die in der Hilfe aufgeführten Beispiele überprüfen.

Präfixe
Gibt die Präfixe an, die für generierte JNDI-Namen verwendet werden sollen.
Überschreiben
Gibt an, ob vorhandene Bindungen überschrieben werden sollen.

Ist das Markierungsfeld ausgewählt, werden die vorhandenen Bindungen von den generierten Bindungen überschrieben.

CMP-Bindungen für EJB 1.1
Gibt den JNDI-Namen der Standarddatenquelle an.

Ist der Radioknopf Standardbindungen für CMPs gemäß EJB 1.1 ausgewählt, müssen Sie den JNDI-Namen für die Standarddatenquelle, die mit CMP-1.1-Beans verwendet werden soll, angeben. Geben Sie auch die Benutzer-ID für diese Standarddatenquelle an.

Bindungen für Verbindungs-Factorys
Gibt den JNDI-Namen der Standarddatenquelle an.

Ist der Radioknopf Standardbindungen für Verbindungs-Factorys ausgewählt, müssen Sie den JNDI-Namen für die Standarddatenquelle, die mit den Bindungen verwendet werden soll, angeben. Geben Sie auch die Berechtigung für die Ressource an.

Virtueller Host
Gibt den virtuellen Host für WAR-Module an.
Spezifische Bindungsdatei
Gibt eine Bindungsdatei an, die die Standardbindung überschreibt.

Sie können das Verhalten der Standardbindung mit einem XML-Dokument (benutzerdefinierte aka-Strategie) ändern. Benutzerdefinierte Strategien erweitern die Standardstrategie, so dass Sie nur die Bereiche, in denen die Standardstrategie unzureichend ist, anpassen müssen. Das heißt, Sie müssen lediglich beschreiben, wie Sie die von der Standardstrategie generierten Bindungen ändern möchten. Es ist nicht erforderlich, Bindungen für die gesamte Anwendung zu definieren.

Nachfolgende kurze Beispiele für das Überschreiben, die verschiedene Bereiche des Generators für Standardbindungen abdecken:

JNDI-Namen einer EJB steuern 

<?xml version="1.0"?>
<!DOCTYPE dfltbndngs SYSTEM "dfltbndngs.dtd">
<dfltbndngs>
  <module-bindings>
    <ejb-jar-binding>
      <jar-name>helloEjb.jar</jar-name>
<!-- this name must match the module name in the .ear file ->
      <ejb-bindings>
        <ejb-binding>
         <ejb-name>HelloEjb</ejb-name>
<!-- this must match the <ejb-name> entry in the EJB jar DD ->
         <jndi-name>com/acme/ejb/HelloHome</jndi-name>
        </ejb-binding>
      </ejb-bindings>
    </ejb-jar-binding>
  </module-bindings>
</dfltbndngs>

Bindung einer Verbindungs-Factory für eine EJB-JAR-Datei definieren 

<!DOCTYPE dfltbndngs SYSTEM "dfltbndngs.dtd">
<dfltbndngs>
  <module-bindings>
    <ejb-jar-binding>
      <jar-name>yourEjb20.jar</jar-name>
      <connection-factory>
        <jndi-name>eis/jdbc/YourData_CMP</jndi-name>
        <res-auth>Container</res-auth>
      </connection-factory>
    </ejb-jar-binding>
  </module-bindings>
</dfltbndngs>

Bindung einer Verbindungs-Factory für eine EJB-Datei definieren 

<?xml version="1.0">
<!DOCTYPE dfltbndngs SYSTEM "dfltbndngs.dtd">
<dfltbndngs>
  <module-bindings>
    <ejb-jar-binding>
      <jar-name>yourEjb20.jar</jar-name>
      <ejb-bindings>
        <ejb-binding>
          <ejb-name>YourCmp20</ejb-name>
<!-- this matches the ejb-name tag in the DD ->
          <connection-factory>
           <jndi-name>eis/jdbc/YourData_CMP</jndi-name>
           <res-auth>PerConnFact</res-auth>
          </connection-factory>
        </ejb-binding>
      </ejb-bindings>
    </ejb-jar-binding>
 </module-bindings>
</dfltbndngs>

Bindung einer Ressourcenreferenz aus einer WAR-, EJB-JAR- oder J2EE-Client-JAR-Datei überschreiben

Nachfolgend ein Beispielcode für das Überschreiben der Bindung einer Ressourcenreferenz aus einer WAR-Datei: Sie können einen ähnlichen Code verwenden, um die Bindung einer Ressourcenreferenz aus einer EJB-JAR- oder J2EE-Client-JAR-Datei zu überschreiben.

<?xml version="1.0"?>
<!DOCTYPE dfltbndngs SYSTEM "dfltbndngs.dtd">
<dfltbndngs>
  <module-bindings>
    <war-binding>
      <jar-name>hello.war</jar-name>
      <resource-ref-bindings>
        <resource-ref-binding>
          <!-- the following must match the resource-ref in the DD ->
          <resource-ref-name>jdbc/MyDataSrc</resource-ref-name>
          <jndi-name>war/override/dataSource</jndi-name>
        </resource-ref-binding>
      </resource-ref-bindings>
    </war-binding>
  </module-bindings>
</dfltbndngs>

MDB-JMS-Listener-Ports überschreiben 

<?xml version="1.0"?>
<!DOCTYPE dfltbndngs SYSTEM "dfltbndngs.dtd">
<dfltbndngs>
  <module-bindings>
    <ejb-jar-binding>
      <jar-name>YourEjbJar.jar</jar-name>
      <ejb-bindings>
        <ejb-binding>
          <ejb-name>YourMDB</ejb-name>
          <listener-port>yourMdbListPort</listener-port>
        </ejb-binding>
      </ejb-bindings>
    </ejb-jar-binding>
  </module-bindings>
</dfltbndngs>
