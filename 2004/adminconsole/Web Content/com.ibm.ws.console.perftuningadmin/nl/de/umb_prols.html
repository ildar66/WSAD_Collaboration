Message Listener Service

Der Message Listener Service ist eine Erweiterung der JMS-Funktionen des JMS-Provider. Der Service bietet einen Listener-Manager, der einen oder mehrere JMS-Listener steuert und überwacht. Die JMS-Listener überwachen ihrerseits jeweils eine JMS-Zieladresse für eine implementierte (Deployment) nachrichtengesteuerte Bean.

Diese Anzeige enthält Links zu den Seiten "Weitere Merkmale" für Listener-Ports, Thread-Pools und "Benutzerdefinierte Merkmale" für den Message Listener Service.

Klicken Sie zum Anzeigen dieser Seite der Administrationskonsole auf Server -> Anwendungsserver -> Nachrichten-Listener-Service.

Passen Sie diesen Parameter an, wenn mehrere nachrichtengesteuerte Beans in demselben Anwendungsserver implementiert werden und die Summe ihrer maximalen Sitzungswerte den Standardwert 10 überschreitet.

Benutzerdefinierte Merkmale
Benutzerdefinierte Merkmale des Message Listener Service. Der JMS-Server kann in zwei Modi arbeiten: Application Server Facilities (ASF) und Non-ASF. ASF unterstützt gemeinsamen Zugriff und Transaktionen in Anwendungen. Non-ASF umgeht diese Unterstützung, um die Pfadlänge zu optimieren.

Verwenden Sie Non-ASF in den folgenden Fällen:

  • Die Reihenfolge der Nachrichten ist eine strikte Anforderung.
  • Gleichzeitige PTP-Nachrichten sind erwünscht.
  • Das Merkmal non.asf.receive.timeout ist angegeben und hat einen Wert größer als 0.
Verwenden Sie Non-ASF nicht, wenn gleichzeitige Pub/Sub-Nachrichten erwünscht sind. ASF bietet hier einen besseren Durchsatz.

Datentyp Integer
Einheiten Millisekunden
Standardwert ASF-Modus (keine Erstellung benutzerdefinierter Merkmale)
Bereich
Empfohlene Einstellung Der Wert steht für die Anzahl Millisekunden, die für die Zustellung einer Nachricht benötigt werden. Wenn das Zeitlimit überschritten wird, muss die Zustellung wiederholt werden. Dies bedeutet zusätzlichen Aufwand. Sie sollten für diese Einstellung einen geringfügig niedrigeren Wert als für das Transaktionszeitlimit wählen. Die Differenz sollte einige Sekunden (10 oder mehr) betragen. Die Differenz sollte bei extrem hohen Arbeitslasten, wenn Threads lange auf die Zuteilung von CPU-Zeit warten, höher gewählt werden.
